# End-to-end QA report

Date: 2026-07-20
Target: `https://creator-match-app-zeta.vercel.app` (production)
Method: real browser session driving the deployed app, signed in, against live Tavily, YouTube,
Gemini and Supabase. No mocks or stubs at any layer.

## Result

Every functional path except Gmail sending was exercised and works. Three defects were found and
fixed, one issue remains open for the operator, and two checks could not be completed.

## Verified working

| Area | Evidence |
|---|---|
| Landing page | Loads, title correct, GSAP entrance animation runs |
| Live search | 50 results returned, 16 strict matches, real subscriber counts and engagement |
| Authentication | Valid `sb-*-auth-token` session; magic-link sign-in confirmed working |
| Dashboard metrics | All four tiles populate; "Creators found" advanced 50 → 100 → 150 across runs, confirming `discovery_runs` logging |
| Save creator | Three creators saved through RLS as a real user, no policy errors |
| Outreach generation | Gemini returned a 79-word email and 24-word DM, both in range, addressing the creator by name and referencing their actual content theme |
| CSV export | 86-row deliverable generated through the same `creatorsToDownload` path, all columns populated |
| RLS enforcement | Anonymous INSERT rejected with `42501`; anonymous SELECT returns `[]` |
| Schema | All six tables present and queryable |

The RLS result is worth stating plainly: the earlier prototype's audit flagged
`USING (true) WITH CHECK (true)` as CRITICAL, meaning anyone holding the anon key could read or
delete all data. The current project does not have that flaw.

## Defects found and fixed

### 1. Engagement rates inflated by an order of magnitude — fixed

Live results reported creators at 20–32% engagement and the dashboard showed a mean of 20.75%,
against a credible YouTube range of roughly 1–8%. Cause: the mean was computed across every sampled
video regardless of view count, so a video with 40 views and 12 interactions read as 30%.

Fix: videos below 100 views are excluded from the sample, and a creator whose entire sample falls
below that floor reports no rate rather than a flattering invented one. Two regression tests added.

After the fix, across 86 qualified creators: median 2.35%, maximum 9.54%.

### 2. Sorting buried every contactable creator — fixed

The result sort applied the chosen metric globally, so profiles with high (inflated) engagement and
no contact email outranked creators who could actually be reached. Saving the top three Fashion
results produced three creators with no contact email — the opposite of the product's purpose.

Fix: strict matches lead in every sort, with the metric applied within each group. After the fix the
top six results were all strict matches with contact emails.

### 3. Sign-in promised a code that cannot be sent — fixed

The UI instructed users to enter a six-digit code. Supabase locks email-template editing unless
custom SMTP is configured, and its default Magic Link template omits `{{ .Token }}`, so on the
built-in email service no code is ever sent. Copy now leads with the magic link and treats the code
field as an optional fallback, which stays correct if custom SMTP is added later.

## Open for the operator

### Stale engagement values on saved creators

Three creators saved during this QA pass were stored *before* the engagement fix, so their rows hold
the old inflated values (31.83%, 20.57%, 9.84%) and the dashboard still reports a 20.75% mean.
Saved rows are snapshots and are not recomputed.

Resolution: delete those three rows from `saved_creators` and re-save them, or clear them entirely.
They are QA artifacts, not real shortlist entries.

## Responsive layout — verified at 375×812

The first attempt used the signed-in browser window, which reported `clientWidth: 0` regardless of
resize because the window was minimised, making every measurement meaningless. Re-run in a
controlled 375×812 viewport instead:

| Surface | Horizontal scroll | Overflowing elements |
|---|---|---|
| Landing | none | none outside the marquee |
| Search + 8 result cards | none | none — cards render 335px inside a 375px viewport |
| `/auth` | none | none — all form controls fit |

The landing page reports eight elements wider than the viewport, all inside `.marquee-track`. That
is the scrolling niche ticker, which must exceed its container to animate and is clipped by an
`overflow-hidden` parent. Page-level `scrollWidth` equals `clientWidth` at 375px on every surface
tested, so there is no horizontal scroll.

## Discarded false positive

The landing hero initially appeared to render at 20% opacity. Investigation showed the GSAP entrance
animation progressing monotonically toward full opacity, throttled because the automated tab was
backgrounded (`visibilityState: hidden`, `requestAnimationFrame` starved). Not a defect.
